# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Repo Is

**MindDeck** — converts XMind mind maps into interactive browser-based product review tools. Zero build steps, zero npm dependencies. Pure HTML5 + Vanilla JS, served by `python3 -m http.server`.

## Key Commands

```bash
# Run integration tests (Node.js VM, no browser needed)
node scripts/test-modules.js

# Parse an XMind file → data.js + screenshots/
node scripts/parse-xmind.js <file.xmind> <output-dir>

# Generate projects.json manifest (scans Projects/)
node scripts/gen-manifest.js <root-dir>

# One-command launch: parse + serve + open browser
bash scripts/launch.sh <file.xmind> [port]

# Verify screenshots referenced in data.js exist
bash scripts/verify-images.sh
```

## Architecture

### Module Load Order (critical — classic `<script>` tags, not ES modules)

```
data.js → storage.js → tree.js → hotspots.js → comments-ui.js
       → screen.js → export-ui.js → ai.js → app.js
```

Each module declares globals with `function` (hoisted) or `const`/`let` (top-level, browser-global but NOT visible in Node.js `vm` sandbox). `storage.js` runs first and sets shared constants (`TREE`, `META`, `PROJECT_ID`, `LS_PREFIX`) that all later modules depend on.

### Module Responsibilities

| File | Owns |
|---|---|
| `data.js` | `window.PROJECT_META`, `window.PROTOTYPE_TREE` |
| `storage.js` | All persistence (localStorage + IndexedDB), image resolution, comments data layer, hotspot positions, export/import |
| `tree.js` | Left sidebar outline tree render |
| `hotspots.js` | Hotspot overlay on screenshots, Shift+drag resize/move with snap guides |
| `comments-ui.js` | Pin-drop comments, popovers, badge counts; owns `COMMENT_MODE` global |
| `screen.js` | Main screenshot area render, breadcrumb, inspector panel |
| `export-ui.js` | Export/import dialogs, toast notifications |
| `ai.js` | Claude Vision hotspot auto-location (tile slicing, candidate review UI) |
| `app.js` | Node index, routing (`selectNode`), search, toolbar wiring, init |

### Storage Layout

- **localStorage**: comments (`proto:<id>:comments_v1`), hotspot positions (`proto:<id>:hs_positions_v1`), image override metadata (`proto:<id>:img_overrides_v1`), last-selected node
- **IndexedDB**: image blobs (per-project DB `proto_imgs__<id>`)
- **Project isolation**: every key is namespaced by `PROJECT_ID` from `META.id`

### Multi-Project Setup

```
UI-model/
  index.html          ← project picker (fetches projects.json)
  projects.json       ← generated by gen-manifest.js
  Template/           ← source of truth for all JS modules + Prototype.html
  Projects/<name>/    ← one directory per project, files copied from Template/
  scripts/
    launch.sh         ← parse XMind → copy Template/ → serve from root → open browser
```

`launch.sh` always copies ALL files from `Template/` to `Projects/<name>/`. **When updating any JS module or Prototype.html, edit both `Template/` AND `Projects/hnw-licai/`** (the reference project). Users re-run `launch.sh` to propagate changes to their own project directories.

### AI Hotspot Flow

1. `autolocateCurrent` → `autolocateNode` (detects, does NOT save) → `showHotspotCandidates` → user confirms → `commitResults` → saves
2. `autolocateAll` (batch) → `autolocateNode` → `commitResults` directly (no review UI)
3. Hotspots with `manual: true` are never overwritten by AI
4. `cleanupCandidates()` is called at the top of `renderScreen` to clear candidate UI on node switch

### `data.js` Schema

```js
window.PROJECT_META = { id: string, title: string, sub: string|null };
window.PROTOTYPE_TREE = {
  uid: 'n0',        // sequential n0, n1, n2... (pre-order)
  depth: 0,
  title: string,    // rawTitle with \n → space
  rawTitle: string, // original XMind title
  note: string|null,
  image: string|null, // filename only, e.g. "abc123.png" → screenshots/abc123.png?v2
  children: [...]
};
```

## Testing Notes

`scripts/test-modules.js` runs all 9 modules in a Node.js `vm` sandbox with a mocked browser environment. Key constraint: top-level `const`/`let` in the sandbox are NOT visible as sandbox properties (unlike in a real browser). Test them indirectly through functions. `function` declarations ARE visible.

When adding new globally-needed functions to any module, add them to the `expectedFunctions` list in `test-modules.js`.

## Common Pitfalls

- **Template/ sync**: JS changes must be applied to both `Template/<file>.js` AND `Projects/hnw-licai/<file>.js`. The Template is the canonical source; hnw-licai is the working reference project.
- **`launch.sh` template path**: hardcoded as `$ROOT_DIR/template` (lowercase). macOS APFS is case-insensitive so `Template/` matches; Linux would not.
- **`window.claude`**: The AI features call `window.claude.complete()` which is injected by the Claude Code environment. Not available in plain browsers or Node.js tests.
- **Import/export schema**: export files must have `schema: 'minddeck'` or `'proto-review'` to be accepted by `applyImport`.
